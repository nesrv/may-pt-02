### Декораторы в Python: формально и по-программистски

**С точки зрения семантики языка** декоратор — это синтаксический сахар над вызовом функции (или другого вызываемого объекта), которая *принимает* объект ниже по стеку декораторов и *возвращает* его замену. Иными словами, это частный случай **функции высшего порядка**: аргумент и результат — вызываемые сущности (callable). Запись `@d` над `def f` эквивалентна присваиванию `f = d(f)` (с поправкой на порядок при цепочке декораторов).

**По-инженерному:** вы «оборачиваете» поведение, не копируя одну и ту же обвязку в каждую функцию. Меньше копипасты — меньше шансов, что правку забудут в одном из десяти мест. Это не «магия», а явная композиция, просто компактно записанная.

---

### Встроенные (и стандартные) декораторы: что имеется в виду

В узком смысле «встроенными» называют конструкции уровня языка (`@property`, `@staticmethod`, `@classmethod`). В учебном модуле ниже — **декораторы стандартной библиотеки** (`functools`, `dataclasses`), которые на практике воспринимаются как «родные»: их не пишут руками на каждый проект, а импортируют и ставят как штамп качества.

---

### `@lru_cache` (`functools`)

**Академически:** кэш с политикой вытеснения **LRU** (Least Recently Used) ограничивает число ключей и при переполнении удаляет *наименее недавно использованные* записи. Для чистых функций с дорогим вычислением и повторяющимися аргументами это даёт **амортизированное** ускорение за счёт пространства.

**По-живому:** если входы часто повторяются — зачем гонять одно и то же по CPU? Пусть результат лежит «на полочке». Не чистая функция с сайд-эффектами + кэш = классический **футган**: закэшировали не то, что надо, и долго ловите «почему у меня в проде старьё».

**Пример идеи кэша** (для факториала на больших `n` без хвостовой рекурсии в Python это скорее антипаттерн по памяти/глубине стека; ниже — *иллюстрация* механизма на малых значениях):

```python
from functools import lru_cache

@lru_cache(maxsize=None)
def fact(n: int) -> int:
    return 1 if n <= 1 else n * fact(n - 1)

print(fact(20))
```

Для «тяжёлых» чистых функций (разбор дерева, DP с повторяющимися подзадачами) `@lru_cache` часто превращает экспоненту в что-то терпимое — без отдельного словаря мемоизации.

**Без кэша** тот же факториал на больших `n` упирается в лимит рекурсии и память; в бою обычно либо итерация, либо `math.factorial`, либо осознанно поднимают `sys.setrecursionlimit` (делайте это только если понимаете последствия для стека).

```python
import sys

# sys.setrecursionlimit(10**6)  # осторожно: это не «бесплатный апгрейд»

def F(n):
    return 1 if n == 1 else n * F(n - 1)

print(F(2024) / F(2020))
```

### `@dataclass` (`dataclasses`)

**Академически:** декоратор генерирует шаблонные методы (`__init__`, `__repr__`, `__eq__`, при необходимости сравнения порядка — `__lt__` и т.д.) на основе аннотированных полей. Это снижает **церемониальный** код (boilerplate) и фиксирует контракт данных в одном месте.

**По-живому:** вместо десяти строк «я написал `__init__` руками и где-то опечатался в имени поля» — объявили поля и поехали. Для DTO, конфигов, событий в пайплайне — must have. Не замена нормальной доменной модели со инвариантами, если логика разрослась: тогда уже класс с методами и явными проверками.

```python
from dataclasses import dataclass

@dataclass
class Person:
    first_name: str
    last_name: str
    age: int
    job: str

anne = Person(
    first_name="Anne",
    last_name="Smith",
    age=40,
    job="software engineer",
)
```

Дополнительно `dataclasses` даёт `field()`, `frozen=True` (иммутабельность), `asdict`/`astuple` — удобно для сериализации и тестов.

### `@singledispatch` (`functools`)

**Академически:** реализация **ad hoc полиморфизма** по типу первого аргумента: одна «точка входа» ведёт к разным зарегистрированным реализациям. Близко по духу к перегрузкам в статических языках, но диспетчеризация динамическая.

**По-живому:** когда `if isinstance(...)` разрастается до «лесенки стыда», `singledispatch` выносит ветвление в отдельные функции — проще читать и покрывать тестами.

```python
from functools import singledispatch

@singledispatch
def fun(arg):
    print("дефолт:", arg)

@fun.register(int)
def _(arg):
    print("ветка int")

@fun.register(list)
def _(arg):
    print("ветка list")

fun("привет")
fun(1)
fun([1, 2, 3])
```

### Несколько декораторов: порядок применения

**Формально:** при

```text
@d1
@d2
def f(...): ...
```

сначала применяется `d2`, затем `d1`, то есть `f = d1(d2(f))` (снизу вверх — как «ближайший к функции обёртывает первым»).

**Интуиция:** внешний декоратор видит уже «прокачанную» внутренним обёрткой функцию.

```python
def h1(func):
    def wrapper():
        return "<h1>\n" + func() + "\n</h1>"
    return wrapper

def div(func):
    def wrapper():
        return "\t<p> " + func() + " </p>"
    return wrapper

@h1
@div
def hello():
    return "Привет"

print(hello())
```

Результат:

```html
<h1>
	<p> Привет </p>
</h1>
```

---

### Параметризованный сценарий (цепочка + аргументы)

Когда внутренним уровням нужны аргументы, сигнатуры `wrapper` и оборачиваемой функции должны согласоваться — иначе получите «всё крашится в рантайме», что типично для первой итерации своего декоратора.

```python
def h1(func):
    def wrapper(title):
        return "<h1>\n" + func(title) + "\n</h1>"
    return wrapper

def div(func):
    def wrapper(title):
        return "\t<p> " + func(title) + " </p>"
    return wrapper

@h1
@div
def hello(title):
    return title

print(hello("Привет"))
```

### Мини-резюме

| Декоратор        | Задача в одном предложении                          |
|-----------------|------------------------------------------------------|
| `@lru_cache`    | Мемоизация чистых функций, LRU при ограниченном кэше |
| `@dataclass`    | Генерация рутины для классов-носителей данных        |
| `@singledispatch` | Ветвление по типу без монолитного `isinstance`     |

Если что-то из этого кажется «магией» — разверните в эквивалент без `@`: станет видно, что это обычная композиция функций, просто записанная лаконично.
